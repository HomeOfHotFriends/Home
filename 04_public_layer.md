
# 04 — Public Layer

The public layer is what the person at the machine encounters.

Not as a user interface bolted onto the system. As a playing surface that is already the execution field.


---

## 1. What the machine looks like from the outside

The machine has:

- two operand planes (experienced as two zones, two truths, two voices)
- a synthesis controller (experienced as the mediating instrument — Big Friendly)
- cartridges (experienced as insertable objects with individual identities)
- slots (experienced as positions that change what a cartridge means)
- a state display (experienced as visible color/light that tells the truth about what the machine currently is)

None of these are decorative. All of them are executable.


---

## 2. Current state colors

The machine's visible surface is its truth system.

| Color | Meaning |
|-------|---------|
| Orange | Host alive and ready; no cartridge loaded |
| Green | Cartridge already in slot at startup; bad or invalid condition |

This list will grow. Future states will carry:

- unsupported relations
- unsafe insertion conditions
- missing dependencies
- routing conflict
- unstable merge state

State comes before spectacle. The machine can stand there and mean something before doing anything.


---

## 3. The boot sequence as the player experiences it

```
Machine off
↓
Power on
↓
Orange  —  "I am here. Nothing is loaded."
↓
Insert cartridge
↓
(11,11) fires  —  ignition
↓
(7,7) fires    —  awakening
↓
Ack            —  system coherent
↓
Pass           —  precedence shifts; machine enters interactive phase
```

This is not startup boilerplate. This is the first sentence the machine speaks.


---

## 4. What the player does

The player:

- inserts and removes cartridges
- arranges slot positions
- presses wake
- chooses pass
- moves switches
- uses Big Friendly to synthesize the two planes

Every one of these is an act of programming.

```
playing  =  scripting
```

The player is not operating a pre-written program. The player is changing the decode conditions under which events become instructions.


---

## 5. What outputs look like from the outside

Finished works do not leave the system. They return as new cartridge objects.

```
output  →  new cartridge  →  reinsertable
```

A committed audio piece becomes an audio cartridge. A committed game state becomes a game cartridge. A committed spatial arrangement becomes a level cartridge.

The ecosystem does not exile finished things. It re-citizens them.

```
outputs are not exile
finished works remain capable of return
```


---

## 6. What the machine is, from where the player stands

The machine is not a tool with a user.

It is a living software ecology with a body in it.

The body's actions — touching, arranging, passing, waking — are already inside the execution field.

The machine does not wait to be used. It is already running. It is already reading.

```
Home  =
a discrete dual-domain machine in which ordered pairs normally define coordinates,
isolated axes reveal identity or state, a control layer conditionally synthesizes
the pair into a temporary operational unity, and cartridge-slot interactions act as
executable scripting through state transitions rather than abstract code alone.
```

Or:

```
playing  =  scripting
zero     =  analytic split
cartridge  =  stateful participant
```
